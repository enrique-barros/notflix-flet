Vistas

Las tres pantallas completas de la app. Cada vista devuelve un ft.Column con todo su contenido. No saben en qué ruta están — eso lo gestiona el router.

profile_view.py — Selección de perfil

Para qué sirve: La pantalla inicial de "¿Quién está viendo?". Muestra cuatro avatares y al pulsar cualquiera navega al home.

PROFILES = [
    ("Ana",      "#E50914", ft.Icons.PERSON),
    ("Carlos",   "#0070f3", ft.Icons.PERSON_2),
    ("Marta",    "#7928ca", ft.Icons.PERSON_3),
    ("Invitado", "#444444", ft.Icons.PERSON_OUTLINE),

Es una lista de tuplas. Cada tupla tiene tres elementos: nombre, color hexadecimal e ícono de Flet. Se recorre con un bucle para crear los avatares.

build_profile_view(page)

views/profile_view.py — build_profile_view()
def build_profile_view(page: ft.Page) -> ft.Column:
    from router import navigate_to   # import dentro de la función

    def select_profile(e: ft.ControlEvent) -> None:
        navigate_to(page, "/home")

    profile_row = ft.Row(
        wrap=True,
        alignment=ft.MainAxisAlignment.CENTER,
        controls=[
            _avatar(name, color, icon, on_click=select_profile)
            for name, color, icon in PROFILES   # list comprehension
        ],
    )

    return ft.Column(
        horizontal_alignment=ft.CrossAxisAlignment.CENTER,
        alignment=ft.MainAxisAlignment.CENTER,
        expand=True,
        controls=[
            ft.Text("NOTFLIX", size=56, color=COLOR_PRIMARY),
            ft.Text("¿Quién está viendo?", size=24),
            ft.Container(height=48),  # espaciador
            profile_row,
        ],
    )
¿Por qué el import está dentro de la función? Para evitar importaciones circulares. Si profile_view importara router al nivel superior, y router importara profile_view, Python entraría en un bucle infinito al cargar los módulos. Poner el import dentro de la función lo retrasa hasta el momento en que se necesita, cuando ambos módulos ya están cargados.

List comprehension

[_avatar(name, color, icon, ...) for name, color, icon in PROFILES] es una forma concisa de crear una lista. Equivale a un bucle for que añade cada elemento. El for name, color, icon in PROFILES desempaqueta cada tupla en tres variables.

def _avatar(name, color, icon, on_click) -> ft.Column:
    return ft.Column(
        horizontal_alignment=ft.CrossAxisAlignment.CENTER,
        controls=[
            ft.Container(
                width=100, height=100,
                border_radius=ft.BorderRadius(8,8,8,8),
                bgcolor=color,
                on_click=on_click,
                ink=True,
                content=ft.Icon(icon, color="white", size=44),
            ),
            ft.Text(name, size=15),
        ],
    )

El buscador — on_search()

views/home_view.py — función de búsqueda
def on_search(e: ft.ControlEvent) -> None:
    query = e.control.value.strip().lower()

    if not query:
        catalog_section.visible = True   # vuelve al catálogo
        search_section.visible  = False
        results_grid.controls.clear()
        page.update()
        return

    found = [
        m for m in all_movies
        if query in m.title.lower()
        or query in m.listed_in.lower()
        or query in m.director.lower()
    ][:40]   # máximo 40 resultados

    results_grid.controls   = [_clickable_card(m, page) for m in found]
    no_results.visible      = len(found) == 0
    catalog_section.visible = False
    search_section.visible  = True
    page.update()    # refresca la pantalla

search_bar = ft.TextField(
    hint_text="Buscar títulos, géneros, directores…",
    on_change=on_search,   # se llama en cada tecla
)

on_change dispara la función cada vez que el usuario escribe una tecla. La búsqueda es en tiempo real, sin botón de confirmar. page.update() es necesario para que Flet redibuje los widgets que han cambiado.

La búsqueda convierte todo a minúsculas con .lower() para que "drama", "Drama" y "DRAMA" den el mismo resultado.

_clickable_card() — separar responsabilidades

views/home_view.py — tarjeta con navegación
def _clickable_card(movie: Movie, page: ft.Page) -> ft.GestureDetector:
    from router import navigate_to
    return ft.GestureDetector(
        mouse_cursor=ft.MouseCursor.CLICK,
        on_tap=lambda _: navigate_to(page, f"/detail/{movie.show_id}"),
        content=build_movie_card(movie, width=160),
    )

build_movie_card no sabe nada de navegación. _clickable_card la envuelve en un GestureDetector que sí sabe. Así el componente visual y la lógica de navegación están separados.

La lambda es una función anónima de una sola línea. lambda _: navigate_to(...) ignora el evento (_) y simplemente llama a navigate_to.

_info_row() — filas de metadatos

views/detail_view.py — _info_row()
def _info_row(label: str, value: str) -> ft.Row:
    return ft.Row(
        spacing=10,
        controls=[
            ft.Text(label, size=13, color=COLOR_SUBTEXT, width=90),
            ft.Text(value or "—", size=13, color=COLOR_TEXT, expand=True),
        ],
    )

value or "—" es una expresión Python: si value es una cadena vacía (que es falsy), devuelve "—". Evita mostrar campos vacíos en la interfaz.

El parámetro width=90 en la etiqueta fija su ancho para que todos los valores queden alineados en la misma columna.
