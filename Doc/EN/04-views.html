Views

The app's three full screens. Each view returns a ft.Column with all its content. They don't know which route they're on — that's managed by the router.

profile_view.py — Profile selection

What it does: The initial "Who's watching?" screen. Shows four avatars and navigates to home when any of them is tapped.

The PROFILES list

views/profile_view.py — profile data
PROFILES = [
    ("Ana",      "#E50914", ft.Icons.PERSON),
    ("Carlos",   "#0070f3", ft.Icons.PERSON_2),
    ("Marta",    "#7928ca", ft.Icons.PERSON_3),
    ("Guest",    "#444444", ft.Icons.PERSON_OUTLINE),
]

It's a list of tuples. Each tuple has three elements: a name, a hex color, and a Flet icon. It's iterated with a loop to create the avatars.

build_profile_view(page)

views/profile_view.py — build_profile_view()
def build_profile_view(page: ft.Page) -> ft.Column:
    from router import navigate_to   # import inside the function

    def select_profile(e: ft.ControlEvent) -> None:
        navigate_to(page, "/home")

    profile_row = ft.Row(
        wrap=True,
        alignment=ft.MainAxisAlignment.CENTER,
        controls=[
            _avatar(name, color, icon, on_click=select_profile)
            for name, color, icon in PROFILES   # list comprehension
        ],
    )

    return ft.Column(
        horizontal_alignment=ft.CrossAxisAlignment.CENTER,
        alignment=ft.MainAxisAlignment.CENTER,
        expand=True,
        controls=[
            ft.Text("NOTFLIX", size=56, color=COLOR_PRIMARY),
            ft.Text("Who's watching?", size=24),
            ft.Container(height=48),  # spacer
            profile_row,
        ],
    )
Why is the import inside the function? To avoid circular imports. If profile_view imported router at the top level, and router imported profile_view, Python would enter an infinite loop when loading the modules. Placing the import inside the function delays it until the moment it's needed, when both modules are already loaded.

List comprehension

[_avatar(name, color, icon, ...) for name, color, icon in PROFILES] is a concise way to create a list. It's equivalent to a for loop that appends each element. The for name, color, icon in PROFILES unpacks each tuple into three variables.

def _avatar(name, color, icon, on_click) -> ft.Column:
    return ft.Column(
        horizontal_alignment=ft.CrossAxisAlignment.CENTER,
        controls=[
            ft.Container(
                width=100, height=100,
                border_radius=ft.BorderRadius(8,8,8,8),
                bgcolor=color,
                on_click=on_click,
                ink=True,
                content=ft.Icon(icon, color="white", size=44),
            ),
            ft.Text(name, size=15),
        ],
    )

The search — on_search()

views/home_view.py — on_search()
def on_search(e: ft.ControlEvent) -> None:
    query = e.control.value.lower().strip()

    if not query:
        catalog_section.visible = True
        search_section.visible  = False
        results_grid.controls.clear()
        page.update()
        return

    found = [
        m for m in all_movies
        if query in m.title.lower()
        or query in m.listed_in.lower()
        or query in m.director.lower()
    ][:40]   # maximum 40 results

    results_grid.controls   = [_clickable_card(m, page) for m in found]
    no_results.visible      = len(found) == 0
    catalog_section.visible = False
    search_section.visible  = True
    page.update()    # refresh the screen

search_bar = ft.TextField(
    hint_text="Search titles, genres, directors…",
    on_change=on_search,   # called on every keystroke
)

on_change fires the function every time the user types a key. The search is real-time, with no confirm button. page.update() is needed for Flet to redraw the widgets that have changed.

The search converts everything to lowercase with .lower() so "drama", "Drama", and "DRAMA" all give the same result.

_clickable_card() — separating concerns

views/home_view.py — card with navigation
def _clickable_card(movie: Movie, page: ft.Page) -> ft.GestureDetector:
    from router import navigate_to
    return ft.GestureDetector(
        mouse_cursor=ft.MouseCursor.CLICK,
        on_tap=lambda _: navigate_to(page, f"/detail/{movie.show_id}"),
        content=build_movie_card(movie, width=160),
    )

build_movie_card knows nothing about navigation. _clickable_card wraps it in a GestureDetector that does. This keeps the visual component and the navigation logic separate.

The lambda is a single-line anonymous function. lambda _: navigate_to(...) ignores the event (_) and simply calls navigate_to.

_info_row() — metadata rows

views/detail_view.py — _info_row()
def _info_row(label: str, value: str) -> ft.Row:
    return ft.Row(
        spacing=10,
        controls=[
            ft.Text(label, size=13, color=COLOR_SUBTEXT, width=90),
            ft.Text(value or "—", size=13, color=COLOR_TEXT, expand=True),
        ],
    )

value or "—" is a Python expression: if value is an empty string (which is falsy), it returns "—". This prevents empty fields from showing in the UI.

The width=90 parameter on the label fixes its width so all values align in the same column.

The cast

views/detail_view.py — cast list
if movie.cast:
    for name in [c.strip() for c in movie.cast.split(",")][:8]:
        cast_col.controls.append(ft.Row(controls=[
            ft.Icon(ft.Icons.PERSON_OUTLINE, color=COLOR_SUBTEXT, size=14),

The CSV stores the cast as "Actor One, Actor Two, Actor Three". It's split by comma, whitespace is cleaned with .strip(), and only the first 8 are taken with [:8]. The if movie.cast: prevents errors if the field is empty.
